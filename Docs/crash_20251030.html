The torque macro defined at //v8/src/builtins/wasm-to-js.tq::WasmToJSWrapper() is responsible for accessing to JS Value from WASM Realm. This macro will count how often is the WasmToJSWrapper run and if too often then tierup, once tierup parallel computing with WASM instruction will cause RENDERER CRASH under multicage configuration.

Parallel computing with WASM does not trigger RENDERER CRASH once I comment out thos codes cause tierup of WASM-TO-JS Wrapper.

3. Problem Analysis

The captured crash point is after the point that the heap is corrupted. Then so

Try to check with *Address Sanitizer

Finally, this not works, the report symbols is not correct. Try debugging by attach debugger onto the renderer process. At the point <2025-11-14 Fri>, some valuable facts:

  • Direct cause is missing of GCSafeCode in Heap::GcSafeTryFindCodeForInnerPointer which defined at //v8/src/heap/heap.cc:7162

    The reason of missing is the address of the looking code object is not within cage of Isolate, which can be verify by print the value of cagebase of isolate and value of address within Heap::GcSafeTryFindCodeForInnerPointer.

Next, I need to find out the place that give the invalid return address.

Finally, I find the problem is cause by wasm code in WasmCodePointerTable. WasmCodes that reside in this table is come from different Isolate space. The WasmCodePointer is a process-wide entity, so is shared by isolates, which cause isolates might access codes that compiled for another isolates which may lead to memory access violations.
