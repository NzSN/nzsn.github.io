Assume Windows Platform.

1. Entries

  • wWinMain (if a windows console app)
  • otherwise main()

both defined at ’//chrome/app/chromeexemainwin.cc’

In the most outer most entry (wWinMain or main on windows platform), environment is setup, for example,

  • Setup CWD (Current Working Directory)
  • Run ’base::CommandLine::Init’ which will initialize the commandline singleton from the environment.
  • Extract process type from commandline.
  • Finally, load chrome.dll via MainDllLoader. Load dll and get the address of symbol ’ChromeMain’ and call from the address of ’ChromeMain’.

2. ChromeMain

At ’//chrome/app/chromemain.cc’.

2.1. Entities

  • ChromeMainDelegate: A type that allow embedder to do extra operations on stages across browser startup.
  • ContentMainParams: Contain params that used by content layer.

2.2. Steps

  • Run ’base::CommandLine::Init’ for commandline initialization.
  • trap into content::ContentMain

ChromeMain more like a stage that prepare for content initialization.

3. ContentMain

At ’//content/app/contentmain.cc’

Initialize ContentMainRunner and let the Runner runs.

4. Steps to startup

4.1. Platform specific

4.2. Common Initliazation

  • Commandline initialize: Process able to get it’s process type from base::CommandLine once it’s initialized. After that system may append some extra arguments to commandline. Also commandline setup for headless mode.
  • Load Mojo DLL if MojoCore DLL provided.

4.3. Process-Type-Specific Main function

4.3.1. Browser-Process

  • Initialize MojoCore.
  • Start browser thread pool.
  • Run BrowserMain.
    • Initialize main runner and run it.
