1. Introduction

We are aim to build up a system that perform test specified by a language.

To achieve this goal, there is some problems we need to solve:

  • What scenes are the system be used ?
  • What is the language look like ?
  • Architecture of the system ?

These problem will be thinked in next several sections.

2. Scenes

The major scene is network device testing, when you update software of your network device then run your test scripts via via the system and waiting the result, the result will give you knowledge about what is not pass the test.

4. Language specification

The system perform test that describe by language is called DAL (DevAutomator Language). Before give a language specification, we need to know what things will be described by this language.

First, let’s look at Concepts within DevAutomator. DevAuto has following two main concepts:

  • Machine
  • Operations

All things that done by DevAuto is to let machines apply some configs, do some operations then query informations from them.

so language that to describe these actions must include these two primitives. And together with performance consideration language should be interpreter as fast as possible.

With this requirements we have language in this form Lang Specification.

5. Components of DevAuto

DevAuto consist of following components:

  • Master
  • ActionDB
  • Executors
  • Duts

Here is an overview of it’s components and relations:

Master read ActionScript from ActionDB then transform script into Instructions. After that execute these Instructions with the execution of Instructions Master will send command to Dut and executors to let them to be configured correctly and do some operations.

Let’s look at master.

5.1. Master

A Master is the core of a DevAuto system. It know what is a script said and also able to execute action that describe by script.

Now a problem is what is the structure of Master look like. First let look at what environments that Master will reside in.

Many different types of machines to coporate with, these machines with different types of interface, these machines support different operations.

so with these characteristics of environments have to make sure:

  • Master able to communicate with majority of machines such switchs, PCs and etc.

So Master’s architecture is (PHASE-1 ARCHITECTURE):

Let’s look at TCDB Connector, it’s responsibility is to get TCs from TCDB. Cause of TCDB maybe a git repository, local directory, etc so TCDB Connector should be able to hide this details from another components of Master. This require there have better exist an abstraction between real structure of Data sources and TCDB Connector.

Next to look at Interpreter, It transform TCs into Jobs that able to be dispatched Dispatcher.

Interpreter First use Transformer to transform a TC described by Python Script into Instructions in PyObj then convert these Instructions into Instructions in C++ and finally create a Job to wrapper theses Instructions into it.

JobManager manage all respect about Job processing, how jobs dispatch, how job result return, whether a job is running well and etc. With subcomponents, Dispatcher and RunnerLake, it able to manage Jobs more easily.

Dispatcher is one of subcomponents of JobManager. It help JobManager to make decisions that how a job to dispatch to Runners. JobManager just need to hand jobs to Dispatcher then back to another works because Dispatcher will handle everything that is need from dispatch Jobs. It check whether a Job’s requirements is meet then to make decision whether to dispatch the Job to RunnerLake.

RunnerLake hold collection of Runners, Runner is an entity to process a Job. RunnerLake’s manipulation is Runner, it create Runner, destruct runner, retrieve runner result. Runner is an abstraction of processing a single Job and It own resources that is required by the Job.
